## 一、设计意图复盘

这是 q8 的"诊断"骨干 spec，承担"由外到内三层排查"的方法论原型——执行 → 市场 → 策略。spec 把医学诊断流程（量体温 → 验血 → CT）映射到量化诊断三问，叙事主轴清晰。这份 spec 的特点是**一份 spec 内嵌三个独立诊断**，跟 q1-q7 那种"一份 spec 一个任务"的写法差异较大。【结构】【教学场景：诊断流程】

**为什么把三个诊断放在一份 spec 而不是三份**：合理推断是叙事一致性——"由外到内"是不可拆分的方法论，拆成三份 spec 学员会失去整体感。但代价是 spec 体积膨胀（56 行）+【任务边界】被冲淡（看完不知道哪些是必做、哪些是可选）。这种"叙事完整性优于任务原子性"的设计权衡，本身值得作为方法点记录——但 spec 应当显式说"这三步必须按顺序执行、不可拆分"，而不是默认。【结构】【简洁】

**诊断 1 / 2 / 3 的内部异构性**：诊断 1 是"对照实验"（理想 vs 实盘），诊断 2 是"状态分类 + 表现对照"，诊断 3 是"参数稳定性测试"——三种完全不同的分析方法被装在同一个 spec 里。每种诊断的输出契约也不同：诊断 1 输出"两条净值曲线 + 滚动落差图 + 数字结论"；诊断 2 输出"状态分类阴影图 + 各状态表现表"；诊断 3 输出"5 行对比表 + 排序对照"。这种异构性让【输出契约具体度】的评估很复杂——单看每个诊断都还行，但组合起来就显得"知识密度偏高"。【精确】【可学习】

**【动机】维度的薄弱**：spec 没解释"为什么是 63 日滚动""为什么是千分之一滑点""为什么对 5 种频率（10/15/21/42/63）测试而不是 4 种或 6 种""为什么按交易日中点切分而不是按时间中点"。这些参数选择都在 notebook 注释里有少量说明，但 spec 主体是裸的。诊断类 spec 比建造类 spec 更需要解释参数选择——因为诊断本身就是在"用参数提问"，参数选错了诊断结果会误导。【动机】【可学习】

**触发条件 / 频率 / 阈值【精确】度评估**：诊断 spec 的【精确】维度核心是"什么时候应该跑诊断"（触发）和"诊断结果如何判定通过 / 失败"（阈值）。这份 spec 没说触发条件——隐含是"Step 1 发现恶化就跑 Step 2"，但没显式。诊断 1 的"通过条件"是"走势平稳没有突然放大"——这是定性判断，没量化阈值。诊断 3 的"通过条件"是"参数排序基本一致"——也定性。这种"诊断结果靠学员肉眼判断"的设计，跟生产级监控的 SLO（如 "P99 < 1ms"）有本质差距。【精确】

## 四、可改进点（带改写示例）

### 改进 1：把"诊断结果判定"显式量化【精确】【验证】

**现状**（诊断 1 第 28-30 行）：
```
- 强调重点是走势是否平稳（没有突然放大），而不是绝对数值
- 打印结论：排除执行问题
```
（"平稳""没有突然放大"是定性判断）

**改写**：
```
- 计算判定指标：
   gap_acceleration = rolling_gap.diff().rolling(63).std()  # 落差变化率的滚动标准差
   max_acceleration = gap_acceleration.max()
- 判定阈值：max_acceleration < 0.005（每 63 日落差变化标准差 < 0.5%）→ 执行稳定
- 触发结论："执行稳定（max_acceleration={X:.3%}），排除执行问题。问题不在这一层。"
- 反例：若 max_acceleration ≥ 0.005，结论改为"执行落差异常放大，需进一步检查滑点 / 佣金模型"
```

让"诊断结果"从定性变量化——这是运营 spec 跟 q1-q7 建造 spec 的关键差异（生产级运营必须可量化）。

### 改进 2：补每个诊断的 spec 触发条件 / 频率【精确】【教学场景】

**现状**：spec 没说"诊断 1 / 2 / 3 应该多久跑一次"。隐含是"Step 1 发现恶化就跑一次"，但没显式。

**改写**（加到上下文段末尾）：
```
触发条件与频率：
- 触发：Step 1 监控发现恶化时段（bad_periods 非空）
- 频率：每月例行跑诊断 1（执行健康检查）；恶化时同时跑诊断 2 / 3（市场 + 策略稳定性）
- 输出：诊断 1 / 2 / 3 都生成图 + 数字结论；恶化复盘报告应包含三段
```

让运营 spec 的【触发 + 频率 + 阈值】三要素显式可见。

### 改进 3：把"Kendall tau 或简单比较"二选一【抽象层次】【精确】

**现状**（spec 第 47 行 notebook 注释里）：
```
# Kendall tau 或简单比较
```
（spec 没出现，但暗示了 AI 在两种方案间选择）

**改写**（加到诊断 3 步骤）：
```
4. 诊断 3——参数稳定性检查：
   ...
   - 判定方法：用 scipy.stats.kendalltau 计算前后两半段夏普排序的 tau 值
   - 阈值：tau ≥ 0.7 → 排序稳定，参数没过拟合
   - tau < 0.7 → 参数对时段敏感，可能需要参数自适应
   - 备注：本课程教学版用简单的"top-1 是否一致"做近似（rank_first[0] == rank_second[0]），
     但生产环境应采用 kendalltau
```

让 AI 不用再"二选一"猜——同时把教学简化版与生产版差异显式化。

### 改进 4：把"诊断 3 对比表"格式显式锁定【同步】【精确】

**现状**（spec 第 46 行）：
```
- 打印对比表：频率、前半段年化、前半段夏普、后半段年化、后半段夏普
```
（notebook 实际用了 `│` 分隔符 / 列宽对齐 / 数字对齐符号，spec 没说）

**改写**：
```
- 打印对比表（格式固定，便于复盘报告复制）：
   `频率   │ 前半段年化  前半段夏普 │ 后半段年化  后半段夏普 │ 排序一致？`
   - 列宽：频率 6 字符，年化 10 字符（含正负号），夏普 10 字符（保留 2 位小数）
   - 分隔符 `│`，表头下方一行 `─` 横线
   - 行顺序按频率从小到大
```

把 notebook 已经实现的"漂亮表格"格式回填进 spec，达到字符级【同步】。

### 改进 5：补诊断段的机械 assert【验证】

**现状**：验证段全是"图正常渲染""结论已打印"，无断言。

**改写**（加到验证段开头）：
```
机械可验证断言（运行后立刻检查）：
   assert detector.market_vol is not None and len(detector.market_vol) > 0, "市场波动率序列为空"
   assert detector.high_vol_mask.sum() > 30, "高波动天数过少（<30），数据可能异常"
   assert "high" in detector.performance_by_state(result_base), "市场状态分类缺失 high 类别"
   assert len(results_first) == 5 and len(results_second) == 5, "诊断 3 应有 5 频率 × 2 半段 = 10 个回测结果"
   assert all(v in dir() for v in ["detector", "high_vol_mask", "market_vol", "HIGH_VOL_LINE"]), "Step 3 依赖变量缺失"
```

特别是诊断 3 的 "5 频率 × 2 半段 = 10 个回测" 是接续链关键节点——失败立刻定位。

## 五、可沉淀的方法点（用于书中方法/模板章节）

> 标签体系（10 个）：【结构】【精确】【可复现】【动机】【验证】【正向】【简洁】【同步】【可学习】【教学场景】

### A. 这份 spec 已经示范的原则

- **【结构】"由外到内"诊断方法论**：执行 → 市场 → 策略 三层排查的医学类比（量体温 → 验血 → CT）—— 比起"自由探索"或"跑一堆指标"，这种结构化诊断方法论本身就是教学产出。值得作为方法章节"运营章 spec 的诊断骨架"原型
- **【动机】场景化的诊断动机**：上下文段开头的"Step 1 发现了多段恶化时段，现在需要搞清楚原因"是优秀的接续型 spec 链开场——清楚标明本 spec 在叙事里的位置（前置 / 后续）
- **【正向】全 "做 X"**：spec 全文无 "不要" / "避免"句式
- **【教学场景：oxq 抽象】MarketStateDetector 一行替代手工状态检测**：跟 spec-01 的 StrategyMonitor 同源——"运维原语"打包思路一致
- **【教学场景】先读源码再写代码（要求第 1 步）**：开头 `阅读以下 oxq 模块的源码` 跟 spec-01 一致——q8 系列的 oxq spec 都遵循这个模板

### B. 这份 spec 修补后可示范的原则

- **【精确】诊断结果必须可量化判定**：当前 spec 用"走势平稳""排序一致"等定性判断；修复后的"max_acceleration < 0.005""kendall tau ≥ 0.7"可作为"运营章诊断 spec 的判定阈值四要素（触发 / 阈值 / 通过 / 失败）"原型
- **【精确】运营章 spec 必须显式触发条件 / 频率**：当前 spec 没说"什么时候应该跑这个诊断"；修复后可作为"周期性 spec 的元数据三件套（触发 / 频率 / 输出形态）"示范
- **【抽象层次】不要在 spec 里给 AI "二选一"**：当前 "Kendall tau 或简单比较" 让 AI 选；修复后明确指定一种 + 备注差异，可作为"spec 是契约不是协商"的反面教材
- **【同步】表格格式必须显式锁定**：当前对比表完全靠 AI 设计——格式漂亮但不可复现；修复后的"列宽 / 分隔符 / 行顺序"显式可作为"复盘报告类输出的格式锁定"示范
- **【验证】接续链关键节点 assert**：当前验证段无断言；修复后的"5 频率 × 2 半段 = 10 个回测"等断言可作为"长接续链 spec 的链节断言"示范（与 spec-01 同方向）

### C. 领域 / 教学场景特殊考量（不通用但必须教）

- **【教学场景】一份 spec 包多个诊断的张力**：q1-q7 的 spec 都是"一份 spec 一件事"，q8 的诊断 spec 是"一份 spec 三件事"——这种叙事完整性优于任务原子性的权衡，是运营章 spec 特有的。spec 应当显式声明"这三步是不可拆分的方法论"，而不是默认。学员仿写"我也要做诊断"时，会面对"要不要拆 spec"的抉择，本课程必须给出指导
- **【教学场景】教学简化版 vs 生产版的双面写法**：诊断 3 用"top-1 一致"近似 Kendall tau，是教学简化；spec 应同时给出"教学怎么写"和"生产怎么写"。这种双面写法在 q8 / q9 这种运营章特别重要——学员从这一章直接迁移到自己的实盘运营，需要知道"哪些是教学简化、不能直接搬到生产"
- **【教学场景】诊断 spec 的"由外到内"模板**：执行 → 市场 → 策略 三层是金融领域诊断的通用框架（其他领域可能是"硬件 → 网络 → 应用"或"代码 → 配置 → 数据"）。本课程把这个框架做成 spec 模板，可作为方法章节"领域诊断方法论嵌入 spec 结构"的示范——这是普通技术教程不教的领域骨架
