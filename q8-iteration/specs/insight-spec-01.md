## 一、设计意图复盘

这是 q8 第一份 spec，也是全书后三分之一"运营心智"的开场——它要把学员从"建一个新功能"的造物者心态，切换到"看一个已有策略的健康状态"的运维者心态。spec 第 7 行那句"用具体指标替代焦虑"是整个 q8 章的总动机，它把"监控"这个抽象词翻译成学员当下能体会的情绪管理需求。这种以情绪 / 场景开场的【动机】写法，跟 q1 那种以"我们要画一条线"开场的目标导向写法不同——它更适合"运营"主题。【动机】【教学场景：运营心智切换】

**为什么三个滚动指标是滚动夏普 / 滚动回撤 / 滚动超额而非别的**：spec 没解释，notebook 在 Step 1 引言里给了类比表（体温计 / 水位线 / 跑步比赛）。spec 这种"只列要画什么、不解释为什么这三个"的写法，是依赖 book + notebook 把"为什么"扛走，spec 自己只负责"做什么"——这是【结构】上有意识的分工，但代价是 spec 单独阅读时【动机】维度被掏空。【结构】【动机】

**为什么 roll_window=63**：spec 第 51 行直接给了 63，没解释。notebook 第 4 cell 注释"约 3 个月"也是一笔带过。63 是金融惯例（约 3 个月交易日数），但学员第一次见这个常量会一脸懵——这是【可学习】的薄弱处。一个简单修补：在常量旁边加 `# 63 ≈ 3 个月交易日`。

**触发条件 / 频率 / 阈值的【精确】度评估**：q8 是周期性运营任务，spec 的【精确】维度该考的是"什么时候应该重新跑这个 spec / 触发告警的阈值是多少"。这份 spec 的告警阈值（恶化时段定义为"滚动夏普 < 0 且持续 > 20 个交易日"）出现在 spec 第 65 行的"验证"段而非"要求"段——这是反模式：阈值是规则的一部分而非验证条件。学员复制 spec 时可能不会把验证段一起读，导致 AI 漏掉这个关键阈值。【精确】【结构】

**输出契约是"图 + 列表"而非"告警 + 报告"**：spec 第 60 行要求"打印每段的起止日期、持续天数、平均夏普"——这是个文本列表式的"准告警"。生产级监控的告警通常涉及通道（邮件 / 微信 / Slack）、严重等级（warning / critical）、抑制机制（同一时段不重复报警）。q8 是教学版本，所以输出契约简化成"打印列表"，但 spec 没说清"这是教学简化版，生产环境怎么扩展"。这种"教学简化但不交代"的处理方式，会让学员以后真做实盘监控时少一个抓手。【教学场景】【精确】

## 四、可改进点（带改写示例）

### 改进 1：把告警阈值从【验证】段挪到【要求】段【精确】【结构】

**现状**（spec 第 67 行验证段）：
```
- 恶化时段列表非空（2022 年应有多个时段）
```
（隐含阈值"滚动夏普 < 0 且持续 > 20 交易日"藏在 oxq 库默认参数里，spec 主体没显式）

**改写**（加到要求第 7 步）：
```
7. 自动检测恶化时段——告警规则显式声明：
   - 触发条件：滚动夏普 < 0 且持续 ≥ 20 个交易日
   - 调用 monitor.bad_periods（在 StrategyMonitor 构造时传入 sharpe_threshold=0.0, min_days=20）
   - 输出格式：每段一行，包含起止日期、持续天数、平均夏普
   - 这是教学简化版的告警；生产环境需要扩展通道（邮件 / 即时消息）和严重等级
```

让运营 spec 的核心特征——【触发条件】+【频率】+【阈值】——在 spec 主体显式可见。

### 改进 2：锁死 end_date，统一与 q1 风格【可复现】

**现状**（spec 第 39 行）：
```
- 起始日期：2021-01-01，结束日期：当前日期
```

**改写**：
```
- 起始日期：2021-01-01
- 结束日期：2026-01-01（本课程数据快照截止日；q8 所有图表与诊断结论以此为基准）
- 注：生产环境监控应改为 today；课程教学场景必须锁日期才能保证学员看到与书一致的"2022 年集中恶化"现象
```

直接呼应 q1 同款问题；同时显式说明"教学锁日期 vs 生产用 today"的张力——这是【教学场景】里独特的同位张力。

### 改进 3：补 roll_window=63 的解释，别让常量裸奔【动机】【可学习】

**现状**（spec 第 51 行）：
```
- 用 StrategyMonitor(result_base, benchmark="510300.SS", roll_window=63) 获取滚动指标
```

**改写**：
```
- 用 StrategyMonitor(result_base, benchmark="510300.SS", roll_window=63) 获取滚动指标
  · roll_window=63：约 3 个月交易日（21 天/月 × 3）。这是行业惯例——足够长以平滑短期噪声，
    又足够短以"近期"的代表性
  · 太短（<20）：每天都报警；太长（>120）：警报来时已经亏了半年
```

让学员第一次见到 63 时就理解"为什么不是 60、90、120"。

### 改进 4：补机械 assert 验证【验证】

**现状**：验证段全是描述性"图正常渲染""列表非空"，没有断言。

**改写**（加到验证段开头）：
```
- 机械可验证断言（运行后立刻检查）：
   assert isinstance(monitor.rolling_sharpe, pd.Series), "滚动夏普应为 Series"
   assert len(monitor.bad_periods) >= 1, "应至少检测到 1 段恶化时段"
   assert "510300.SS" in result_base.benchmark_prices, "基准价格未注入"
   assert all(v in dir() for v in ["equity", "daily_ret", "result_base", "monitor"]), "后续 spec 依赖变量缺失"
```

后续 spec 02 / 03 依赖 `equity / daily_ret / result_base / monitor / detector / high_vol_mask` 这条接续链，断言可以让 AI 在 cell 失败时立刻知道"是哪个变量没定义"。

### 改进 5：图表颜色 / 路径 / 轴标签显式化【精确】【同步】

**现状**：spec 没指定颜色、不要求保存图片，但 notebook 用了 #3498DB / #2ECC71 / #E74C3C / #9B59B6 配色 + plt.savefig 到 book/images/。

**改写**（加到要求第 6 步）：
```
6. 构建监控仪表盘：
   ...
   - 配色规范：净值 #3498DB（蓝）、滚动夏普 #2ECC71（绿）、回撤 #E74C3C（红）、超额 #9B59B6（紫）
   - 子图 X 轴共享，最后一行画日期标签；Y 轴标签如规范所列（"净值（起点=100）" 等）
   - 保存图到 ../book/images/01-monitoring-dashboard.png（dpi=150, bbox_inches='tight'）
```

把 notebook 已经做出来的视觉细节回填进 spec，达到字符级一致。

## 五、可沉淀的方法点（用于书中方法/模板章节）

> 本节是本份 review 给方法/模板章节贡献的"原材料"。
> 标签体系（10 个）：【结构】【精确】【可复现】【动机】【验证】【正向】【简洁】【同步】【可学习】【教学场景】

### A. 这份 spec 已经示范的原则

- **【结构】严格四段式 + 先读源码**：上下文 / 任务 / 要求 / 验证；要求第 2 步用 `阅读以下 oxq 模块的源码` 显式触发【先读后写】原则——这是 q2+ 各章涉及 oxq 时的标配开场（值得作为方法章节"oxq 系列 spec 的开场模板"原型）
- **【动机】场景化开场**：上下文用"上个月赚 3% 这个月亏 5%——是正常波动还是策略坏了"这种学员能立刻代入的情绪场景开场，比"为了监控策略健康"这种目标导向写法更具张力——q8 / q9 这种运营章特别适合此写法
- **【正向】全部"做 X"**：spec 全文无 "不要" / "避免"——q8 监控类 spec 容易踩"不要漏报""不要误报"等否定式陷阱，这份没踩
- **【教学场景：oxq 抽象】用 StrategyMonitor 一行替代手工滚动计算**：spec 不要求学员（也不要求 AI）自己 reimplement rolling sharpe / drawdown / excess return。oxq 把这套"运维原语"打包了——这种"框架抽象掉重复计算"的思路，符合本课程"学员不写代码"的核心承诺
- **【简洁】oxq 模块清单显式列出而非"按需导入"**：spec 第 17-26 行把要导入的 oxq 模块全部列出来——比让 AI 自己揣摩"应该导入啥"可控得多

### B. 这份 spec 修补后可示范的原则

- **【精确】告警 spec 必须显式声明触发条件 / 频率 / 阈值 / 输出格式四要素**：当前 spec 把"恶化时段定义"藏在 oxq 默认参数里，修复后可作为"运营类 spec 的告警四要素模板"原型——这是 q8 / q9 这种运营章特有、前 7 章不需要的方法点
- **【可复现】教学锁日期 vs 生产 today 的同位张力**：当前 spec 用 end=today 是隐性时间炸弹；修复后的"两段式说明"（教学锁、生产开）可作为"运营 spec 的双面写法"示范
- **【验证】接续链 assert 替代描述性"已定义"**：当前验证段说"`equity / daily_ret / result_base / monitor` 变量已定义（后续 spec 使用）"是描述性的；修复后的 assert 可作为"接续型 spec 链的链节断言"示范——后续 spec 02 / 03 依赖这些变量，断言失败时 AI 立刻定位
- **【动机】关键常量旁边写"为什么这个数"**：当前 roll_window=63 / 阈值 1.3 中位数 / 20 天最小持续都是裸奔；修复后可作为"参数选择必须解释，不只是声明值"的运营场景示范（q1 已有同样方法点；本份是【动机】维度证据再加一例）

### C. 领域 / 教学场景特殊考量（不通用但必须教）

- **【教学场景】运营章 spec 的"心智切换"开场**：q1-q7 的 spec 都是"建一个新功能"，q8+ 的 spec 是"看 / 修一个已有功能"。spec 第一行的【上下文】要明确告知学员"这个 spec 不要求你产出新功能，而是诊断 / 调整已有功能的运行状态"——这种心智切换不显式说，学员会拿建造者的姿态去做运维，结果就是"看了 30 个图也没看懂在干嘛"
- **【教学场景】告警的"教学简化"vs"生产形态"**：教学版的输出契约是 "print 一个列表"；生产版需要邮件 / 即时消息 / 严重等级 / 抑制机制。spec 应当显式声明"这是教学简化版"——避免学员把教学 spec 直接搬到生产环境
- **【教学场景】oxq 模块清单显式列出**：q8 spec 的 oxq import 比 q1 的 yfinance + matplotlib 复杂得多。把要导入的模块在【要求】段提前列出（比如这份 spec 第 17-26 行 + 第 30 行），而不是让 AI 自己揣摩——这是 q2+ oxq 章节相对通用 prompt 工程的额外【精确】要求
