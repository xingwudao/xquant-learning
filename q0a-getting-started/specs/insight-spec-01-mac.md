## 一、设计意图复盘

这是全课程的"门槛 spec"——决定学员能否跑通后续所有章节。它不是面向 AI 的"做某件事"型 spec，而是面向 AI 的"装好一台机器"型 spec。任务覆盖四件事：装 uv、建 venv、装依赖、跑 check_env.py。这四件事不是线性 task list，而是**前置依赖链**——任何一步静默失败都会让后续章节集体崩盘。spec 用"第零/一/二/三/四步"显式标号承认了这条链。【教学场景】【结构】

**为什么把 check_env.py 直接内嵌而不是让 AI 现写**：环境配置类 spec 的"成功"无法靠图表肉眼对照，必须靠机械断言。如果让 AI 自由生成检查脚本，每次跑出来检查项不同、表达不同、阈值不同——下一章节出错时学员根本不知道是"上一章 check 没覆盖"还是"这一章新问题"。直接给完整源码 = 把 oracle 锁死。这是环境配置 spec 与一般 spec 最大的差异点。【验证】【可复现】

**为什么用 uv 而不是 conda / pyenv + pip**：uv 一条命令搞定 Python 版本管理 + venv + 包安装，把"0 编程基础学员最容易卡壳的三个动作"合并成一个工具。spec 没解释这层，是【动机】缺位——学员不知道"为什么不用我同事推荐的 conda"。

**为什么数据源连通性测试用 akshare 而非 yfinance**：akshare 走国内直连，yfinance 需要科学上网。这份 spec 把"国内网络可达性"作为最后一道闸门——测 akshare 通了说明真的能开课。这个选择背后是项目"国内网络适配"的硬约束，但 spec 没显式说出来。【动机】

**克制的边界**：没有教学员怎么打开 TRAE、怎么注册 GitHub、怎么装终端模拟器——这些在前置环节假设已完成。spec 把自己的领地锁在"终端能用 + 目录已开"之后到"check_env.py 全绿"之前，边界干净。【简洁】

## 二、对照实际可执行性（无 notebook）

环境配置类 spec 的 oracle 不是 notebook 产物，而是**学员能不能从零跑通**。逐项核对：

| 关键执行点 | spec 是否覆盖 | 评价 |
|---|---|---|
| 沙箱外执行标注 | ✓ 首行 `> 所有命令在沙箱外运行` + 第二段二次强调 TRAE 弹窗选择 | 双层提示，符合 spec 写作指南"沙箱处理"规范 |
| 当前工作目录获取 | ✓ `pwd` | 落实"不硬编码路径"原则 |
| Python 版本锁定 | ✓ `uv venv --python 3.12` + check_env 校验 (3,12) | 双重锁，配合 check 脚本可机械验证 |
| 依赖版本锁定 | ✗ `uv pip install ... pandas numpy ...` 无版本号 | **可复现性硬伤**——半年后 pandas 3.0 可能 break notebook |
| open-xquant 安装 | ✓ git+https 直连 + 失败提示完整命令 | 但同样不锁 commit hash / tag |
| check 脚本验证手段 | ✓ check_env.py 6 个检查项 + 失败提示具体修复命令 | 业内做得很好的 validation oracle |
| 错误恢复路径 | ✓ "请把输出结果发给 AI 助手" + 注意事项 3 条 | 落实"错误恢复路径"原则 |
| source $HOME/.local/bin/env | ✓ 安装后立即给出，且注意事项中再提醒 | 防"刚装完 uv: command not found"的常见痛点 |
| 重开终端的提醒 | ✓ "每次打开新的终端窗口，都需要先激活虚拟环境" | 关键的持续运维提醒 |
| 与 Windows 版的对称性 | ✗ Windows 版没有 `source ... env` 这一步，第零步用 `cd 或 echo %cd%` 双命令而 mac 只用 `pwd` | 两份 spec 步骤命名一致，但**激活虚拟环境后 mac 没让学员"再开新窗口验证"**，Windows 版反而要求"关闭并重新打开终端"，对称性破缺 |

**对照结论**：作为一份"机器配置说明书"，覆盖度高（9/11 项达标），但**依赖版本不锁** + **跨平台不对称**两点会在课程发布后第 6-12 个月集中爆发问题。

## 四、可改进点（带改写示例）

### 改进 1：套上 q1 四段式骨架【结构】【可学习】

**现状**：用"第零/一/二/三/四步"自定结构，与课程其他 spec 完全不一致。学员看完第一份 spec，脑子里没有"上下文/任务/要求/结果呈现"模板。

**改写**：保留"第 N 步"作为【要求】内部小标题，外层套四段：

```
**上下文**
学员刚在 TRAE 中打开了一个空目录作为课程学习根目录。本机尚未安装 Python / uv / 任何依赖。这是课程的第一份 spec。

**任务**
在当前目录下完成 macOS 量化课程编程环境配置：装 uv、建 Python 3.12 虚拟环境、装 7 个依赖、跑通 check_env.py 自检。

**要求**
（原"第零~四步"内容，改为编号列表）

**结果呈现**
1. 终端运行 `python check_env.py` 输出 `结果: 9/9 全部通过!`
2. 退出码为 0
3. 当前目录下存在 `.venv/` 与 `check_env.py`
```

第一份 spec 就要让学员看到课程的结构模板，否则第二份开始要纠正习惯成本极高。

### 改进 2：锁死依赖版本【可复现】

**现状**（第三步）：
```
uv pip install --index-url https://pypi.tuna.tsinghua.edu.cn/simple jupyter pandas numpy matplotlib akshare yfinance
```

**改写**：
```
uv pip install --index-url https://pypi.tuna.tsinghua.edu.cn/simple \
  jupyter==1.1.1 pandas==2.2.3 numpy==2.1.3 matplotlib==3.9.2 \
  akshare==1.15.* yfinance==0.2.50
uv pip install "open-xquant @ git+https://github.com/xingwudao/open-xquant.git@v0.1.0"
```

数据源类库 (akshare) 接口变化频繁，至少锁到 minor。open-xquant 必须锁 tag——课程示例代码与框架版本绑定。

### 改进 3：补"为什么"的动机段【动机】【教学场景】

**现状**：第一步只说 "uv 是一个 Python 包管理工具"。

**改写**（在第一步开头加一段）：
```
为什么用 uv：传统方案需要分别学 Homebrew 装 Python + venv 建虚拟环境 + pip 装包共三个工具。uv 一个工具覆盖所有，对 0 编程基础学员把"装环境"的认知负担从 3 个工具降到 1 个。
```

让学员单读 spec 就能理解"为什么不是 conda"，不依赖书侧边栏。

### 改进 4：mac/Windows 双 spec 对称性硬约束【同步】【教学场景】

**现状**：mac 第零步只用 `pwd`，Windows 用 `cd 或 echo %cd%`；mac 装完 uv 后 `source $HOME/.local/bin/env`，Windows 让学员"关闭并重新打开终端"。同一逻辑两种表述。

**改写**：把对称表统一进 spec 头部：

```
| 步骤 | macOS | Windows |
|---|---|---|
| 看当前目录 | `pwd` | `cd` |
| 装 uv 后激活 PATH | `source $HOME/.local/bin/env` | 关闭重开终端 |
| 激活 venv | `source .venv/bin/activate` | `.venv\Scripts\activate` |
```

把 mac/Windows 差异显式列表，两份 spec 共享同一段，差异最小化、可视化。

### 改进 5：check_env.py 抽出独立文件，spec 用引用【简洁】【同步】

**现状**：check_env.py 130 行内嵌在 spec 主体，spec 总长 191 行。后续如果要修改检查逻辑（比如 q2 加 oxq 子模块检查），spec 与脚本两份要同步改。

**改写**：
```
请在当前目录下创建文件 check_env.py，内容从课程仓库 q0a-getting-started/scripts/check_env.py 复制（或让 AI 助手按以下规格生成）：
- 检查 Python >= 3.12
- 检查虚拟环境已激活
- 检查 7 个依赖包导入：jupyter, pandas, numpy, matplotlib, akshare, yfinance, oxq
- 调用 akshare.fund_etf_hist_em(symbol="510300", period="daily", adjust="qfq") 验证数据源连通
- 全绿输出 `结果: N/N 全部通过!` 退出码 0；否则退出码 1
```

spec 体量降一半，脚本独立维护，跨平台一份脚本服两端。

## 五、可沉淀的方法点（用于书中方法/模板章节）

> 标签体系（10 个）：【结构】【精确】【可复现】【动机】【验证】【正向】【简洁】【同步】【可学习】【教学场景】

### A. 这份 spec 已经示范的原则

- **【验证】环境配置类 spec 必须自带 check 脚本**：check_env.py 6 项机械断言 + 退出码 + 失败时具体修复命令。可作为"任何环境/配置类 spec 都必须有 oracle"的原型示范——比 q1 数据获取 spec 的 assert 更彻底（一份独立脚本而非散落断言）
- **【正向】全部"运行 X / 创建 X"**：无 "不要" 句式，符合"做 X 优于不要 Y"原则。可作为该原则的低位锚定示范
- **【教学场景】沙箱外执行的双层提示**：首行 markdown 引用 + 第二段自然语言提醒"TRAE 弹窗选允许"。这是 vibe coding 教学的特殊指令模式，普通 prompt 工程不需要。可作为"AI IDE 沙箱处理"模板单独抽出
- **【教学场景】错误恢复路径写进 spec 而非交给学员**：注意事项 + "把输出发给 AI 助手"贯彻"学员不调试"承诺。可作为"错误恢复路径"原则的最强示范

### B. 修补后可示范的原则

- **【结构】环境类 spec 也要套四段式骨架**：当前用"第 N 步"自定结构是反例；修复后可作为"无论 spec 内容多特殊，外层结构必须统一"的原型 before/after 教学
- **【可复现】依赖版本必须锁到 minor**：当前 `uv pip install pandas` 是反模式；修复后可作为"任何包管理 spec 都不能省版本号"的原型示范——这是【可复现】原则在环境类 spec 上的具体落点
- **【动机】"为什么用 uv"必须写进 spec**：当前只声明"是包管理工具"；修复后可作为"工具选择必须解释，否则学员仿写时换错工具"的示范
- **【同步】跨平台对称表统一管理**：当前 mac/Windows 两份 spec 步骤名一致但命令分散描述，对称性靠人眼对照；修复后可作为"差异点抽对称表 = 跨平台 spec 维护成本最低"的示范
- **【简洁】长嵌入脚本应抽出独立文件**：当前 check_env.py 内嵌占 spec 一半篇幅；修复后可作为"spec 内嵌大段代码是同步噩梦（反模式 12）"的具体救治方案

### C. 领域 / 教学场景特殊考量（不通用但必须教）

- **【教学场景】环境配置类 spec 的特殊性**：与 q1 起的"任务执行类 spec"不同，环境配置类 spec 没有 notebook 产物可对照、没有图表可肉眼判断、错误隐式后传。**oracle 必须是机械可执行的检查脚本**——这是该类 spec 与其他类 spec 最大的差异，方法章节必须单列"环境/配置类 spec 写法"小节
- **【教学场景】跨平台 spec 的对称性硬约束**：mac/Windows 双支持是本课程硬约束。两份 spec 步骤命名、检查项、退出条件必须**字面对称**，差异只能在"具体命令"层。两份不一致本身就是教学反例——学员第一次看到课程的双平台 spec，对称破缺会让其失去"两份 spec 是同一任务"的心智模型。方法章节应给"对称性 checklist"
- **【教学场景】"门槛 spec"的失败放大效应**：本份 spec 是后续 35 份 spec 的前置依赖。这里 1 个静默失败 = 后续 35 份课程的体验崩塌。门槛 spec 的【验证】权重应高于其他 spec，方法章节应承认这种"位置敏感性"——不是所有 spec 都同等重要
- **【教学场景】uv / Python 3.12 是项目硬约定**：spec 写作指南里明确写死"uv + Python 3.12"。本份 spec 必须严格遵守，不能"偶尔用 conda 也行"留口子——版本号、工具名、命令字面级精确，是【精确】原则在环境类 spec 的最强落点
